# paper-distiller v1.6 — Local arxiv Metadata Mirror

**Status:** draft
**Goal:** Stop hitting arxiv's live API for search; mirror their metadata locally so the agent never rate-limits us again. PDF download stays on-demand.
**Anchor commit:** `bb08924` (v1.5.x main)

---

## Why

paper-distiller v1.5.x had three rounds of rate-limit pain:
- Concurrent search calls tripped arxiv's IP-level cooldown (10-60 min)
- SS / OpenAlex degradation cascaded
- Auto-fallback helped but didn't eliminate cold-query latency

**Root cause:** every cold query goes to arxiv's `export.arxiv.org/api/query` — the same endpoint TOS-limits 1 req / 3s. As soon as the agent fires 2-3 cold queries in a minute we're cooked.

**Real fix:** mirror arxiv's metadata locally. arxiv officially supports this via:
- **OAI-PMH protocol** at `https://export.arxiv.org/oai2` (1000 records / 15s — different limit from the search API)
- **Bulk metadata dump** updated weekly on Kaggle + mirrored to Internet Archive

After bootstrap (~30 min one-time), search becomes 100% local. We never touch live arxiv except for individual PDF downloads — and those are rare (only when user explicitly distills), respect the SourceLimiter, and never reach the burst pattern that triggers blocks.

This is the established pattern: paper-search-mcp, OpenReview tooling, FutureHouse's metadata enrichment all use bulk-dump + incremental-OAI.

---

## Scope

**In scope (v1.6):**
- Bulk metadata mirror (~1.7M arxiv papers) in local SQLite + FTS5
- One-time bootstrap from Internet Archive (no auth) + Kaggle fallback + OAI-PMH from-scratch fallback
- Daily incremental sync via OAI-PMH (`from=<last_sync>`)
- Local full-text search via SQLite FTS5 (title + abstract, BM25 ranking)
- `ArxivSearcher` switches to **local-first** with optional live-API fallback for very recent papers
- New CLI entry point `paper-distiller-arxiv` (bootstrap / sync / search / doctor / stats)

**Out of scope:**
- PDF caching (still on-demand via existing pipeline)
- Semantic embedding / vector search (FTS5 keyword is enough for ranker handoff)
- SS / OpenAlex mirroring (different problem; their bulk-data story is worse)
- Auto-scheduled cron sync (user runs `paper-distiller-arxiv sync` manually for v1.6; cron in v1.7)

## SQLite schema

```sql
PRAGMA journal_mode = WAL;        -- multi-process safe
PRAGMA synchronous = NORMAL;      -- fast bulk inserts
PRAGMA cache_size = -64000;       -- 64 MB page cache

CREATE TABLE papers (
  arxiv_id    TEXT PRIMARY KEY,        -- e.g. "2401.12345" or "cs/0205071"
  title       TEXT NOT NULL,
  authors     TEXT NOT NULL,           -- JSON ["Author 1", "Author 2"]
  abstract    TEXT NOT NULL,
  categories  TEXT NOT NULL,           -- JSON ["cs.LG", "stat.ML"]
  primary_category TEXT,
  published   TEXT NOT NULL,           -- ISO date
  updated     TEXT,
  doi         TEXT,
  comment     TEXT,                    -- arxiv comment field
  journal_ref TEXT,
  fetched_at  TEXT NOT NULL,
  source      TEXT NOT NULL            -- 'bootstrap' | 'oai-pmh' | 'live-api'
);

CREATE INDEX idx_papers_primary_category ON papers(primary_category);
CREATE INDEX idx_papers_published ON papers(published);
CREATE INDEX idx_papers_doi ON papers(doi) WHERE doi IS NOT NULL;

CREATE VIRTUAL TABLE papers_fts USING fts5(
  title, abstract,
  content='papers',
  content_rowid='rowid',
  tokenize='porter unicode61 remove_diacritics 2'
);

-- Keep FTS in sync with papers on insert/update/delete
CREATE TRIGGER papers_ai AFTER INSERT ON papers BEGIN
  INSERT INTO papers_fts(rowid, title, abstract)
  VALUES (new.rowid, new.title, new.abstract);
END;

CREATE TRIGGER papers_ad AFTER DELETE ON papers BEGIN
  INSERT INTO papers_fts(papers_fts, rowid, title, abstract)
  VALUES('delete', old.rowid, old.title, old.abstract);
END;

CREATE TRIGGER papers_au AFTER UPDATE ON papers BEGIN
  INSERT INTO papers_fts(papers_fts, rowid, title, abstract)
  VALUES('delete', old.rowid, old.title, old.abstract);
  INSERT INTO papers_fts(rowid, title, abstract)
  VALUES (new.rowid, new.title, new.abstract);
END;
```

**Why FTS5 over external embedding search:**
- 1.7M papers fit in <5 GB with FTS5; vector index would be 10× larger
- BM25 ranking is "good enough" — CandidateRanker (LLM) re-ranks top-50 anyway
- No new heavy dep (sentence-transformers, faiss, etc.)
- Phase 2/3 can add semantic search if BM25 quality is a real bottleneck

## Bootstrap

`bootstrap.py` downloads + parses ~1 GB compressed JSONL → SQLite, in this order:

1. **Internet Archive** (`https://archive.org/download/arxiv-bulk-metadata/...`) — no auth, fast CDN
2. **Kaggle** (`Cornell-University/arxiv`) — needs `~/.kaggle/kaggle.json`; tried only if user set `PD_BOOTSTRAP_SOURCE=kaggle`
3. **OAI-PMH from-scratch** (~6-7 hours, 1000 records / 15s) — final fallback, no auth, slow

```python
def bootstrap(source: str = "auto", dest_dir: Path = DEFAULT_DIR) -> BootstrapResult:
    if source == "auto":
        for src in ["internet_archive", "kaggle", "oai_pmh"]:
            try:
                return _bootstrap_one(src, dest_dir)
            except BootstrapError as e:
                log.warn(f"{src} failed: {e}; trying next")
        raise BootstrapError("all bootstrap sources exhausted")
    return _bootstrap_one(source, dest_dir)

def _bootstrap_one(source, dest_dir):
    # 1. Download dump (JSONL.zip)
    # 2. Stream-parse, batch insert into papers table
    # 3. Build FTS5 index incrementally (triggers handle it)
    # 4. Write state.json with last_sync = dump's snapshot_date
    # 5. Delete dump.tmp
```

**Streaming parse**: JSONL is line-delimited so we never load the whole file in memory. Insert in batches of 10,000 (~50 MB transactions) for SQLite throughput.

Bootstrap is **idempotent**: re-running on an existing DB skips already-present arxiv_ids (UPSERT on PRIMARY KEY).

## Incremental sync via OAI-PMH

`incremental.py` wraps the [Sickle](https://pypi.org/project/sickle/) library:

```python
def sync(since: str | None = None) -> SyncResult:
    """Pull all papers updated since `since` (ISO date) via OAI-PMH.

    arxiv's OAI server gives 1000 records per resumption-token request,
    with a 15s implicit delay between batches. Sickle handles resumption
    automatically; we just iterate.
    """
    since = since or state.last_sync
    sickle = Sickle("https://export.arxiv.org/oai2")
    records = sickle.ListRecords(
        metadataPrefix="arXiv",
        **{"from": since},  # arxiv's OAI accepts ISO date or YYYY-MM-DD
    )
    n = 0
    for record in records:
        store.upsert(record.to_paper())  # parsing in oai_record.py
        n += 1
        if n % 1000 == 0:
            state.checkpoint(record.datestamp)  # resume-safe
    state.last_sync = datetime.utcnow().isoformat()
    return SyncResult(n_new=n_new, n_updated=n_updated, duration_sec=...)
```

**Resume-safe**: state.json is checkpointed every 1000 records, so an interrupted sync picks up where it left off.

**Rate**: respects arxiv's 1000/15s — Sickle's built-in handling + no parallelism.

## Local FTS5 search

`search.py` exposes a single function the searcher uses:

```python

        SELECT p.* FROM papers p

        {category_clause} {since_clause}
        ORDER BY {order_clause}
        LIMIT ?
    """
    # ... build clauses, execute, hydrate to Paper objects
```

**BM25 vs date sort**: when `sort="date"`, we order by `p.published DESC` instead of FTS5 bm25 score. The MATCH filter still applies so we get "newest papers matching X".

**Edge cases handled:**
- Empty query → return [] (don't waste a full-table scan)
- Query has FTS5 operators (`OR`, `NEAR`) → either honor (advanced users) or escape (default); add a `raw_fts=True` param for power users
- Cross-listed papers (multiple categories): JSON array in `categories` column, primary_category for indexed filtering

## Fetcher abstraction

`fetcher.py` defines a small interface so live API / local DB / hybrid all look the same to the searcher:

```python

    def search(self, query: str, n: int, sort: str) -> list[Paper]: ...
    def is_available(self) -> bool: ...

    """Backed by SQLite FTS5. Always available if DB exists."""
    def search(self, query, n, sort):
        return local_search.search(query, n, sort)
    def is_available(self):
        return store.db_exists() and store.paper_count() > 0

class LiveFetcher:
    """Backed by export.arxiv.org via existing arxiv lib + SourceLimiter."""
    def search(self, query, n, sort):
        # ... existing arxiv_search logic
    def is_available(self):
        return not ARXIV_LIMITER.is_cooling_down()

    def __init__(self, local, live, allow_live_topup=True, min_local_ratio=0.5):
        ...

        if (
            len(local_results) >= n * self.min_local_ratio
            or not self.allow_live_topup
            or not self.live.is_available()
```

`ArxivSearcher.run()` instantiates `LocalFirstFetcher` and calls `.search()`. The rest of the DAG is unchanged.

`PD_ARXIV_LOCAL_ONLY=1` env disables the live-topup path entirely — useful for offline / air-gapped use.

## Edge cases & risk mitigation

| Risk | Mitigation |
|---|---|
| User skips bootstrap → local DB empty | LocalFirstFetcher falls through to live API; works exactly like v1.5 |
| Internet Archive mirror down | Auto-fallback chain to Kaggle → OAI-PMH from-scratch |
| Kaggle API rate limit during bootstrap download | Single big download; not a rate-limit candidate |
| OAI-PMH gives weird records (deleted, withdrawn) | Skip + log; don't insert |
| User's disk is full mid-bootstrap | Detect ENOSPC, clean up dump.tmp, fail with clear message |
| Bootstrap interrupted | dump.tmp + state.json checkpointing; re-run resumes |
| Schema migration in v1.7 | `state.schema_version` field; migrate on startup |
| WAL mode + multi-process | Standard SQLite WAL handles concurrent readers + one writer; documented |
| Bootstrap dump is stale (snapshot from 2 weeks ago) | First sync after bootstrap pulls the gap via OAI-PMH from snapshot_date |
| `cs/0205071`-style old IDs | Schema treats arxiv_id as TEXT; both formats stored |
| FTS5 doesn't tokenize CJK well | Out of scope for v1.6 — arxiv abstracts are English; revisit if user complains |
